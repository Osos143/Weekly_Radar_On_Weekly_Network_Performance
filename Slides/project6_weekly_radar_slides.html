Dataset & Filtering

What was the dataset?

The source workbook was a mobile network performance dataset with 93,364 rows. It mixed daily and weekly records, contained both locality-level rows and a country aggregate, and covered multiple regions, carriers, and technologies.

1) Removed daily rows
We kept weekly rows only so that all comparisons used the same time granularity.
2) Kept locality rows
The main workflow used weekly locality rows because they are the correct starting level for aggregation.
3) Separated country rows
Country aggregate rows were moved out of the main analysis table and kept only as benchmark rows.
4) Aggregated by region
Locality rows were aggregated into a clean region × carrier × technology × week table using sample_count-weighted averages.
5) Applied a sample threshold
We removed weak groups below sample_count_total = 30 to keep reliability while preserving full regional coverage.
6) Added helper columns
Helper columns were added to support mean vs. median comparison and later anomaly analysis.

Lessons Learned
1) Preprocessing is part of the model
The biggest improvement came before anomaly detection: removing daily rows, separating country aggregates, aggregating localities with sample-count weighting, and choosing a balanced threshold of 30.
2) Cross-sectional comparison was the right fit
With only 3 weekly snapshots, comparing each carrier against same-week, same-region, same-technology peers gives stronger evidence than forcing a time-series model. Complete peer groups gave 330 valid rows.
3) Why not long-history models first?
  • Rolling z-score / EWMA needs many historical weeks.
  • LSTM or forecasting models need enough history to learn normal behavior.
  • Example: with only 3 weeks, a sudden drop cannot be separated from normal seasonality.
4) Interpretability matters
A good anomaly record should answer: which region, which carrier, which technology, which week, which KPI gap, and which threshold caused the alert.
Key technical takeaway
Cross-sectional analysis is strongest when the dataset has few time points but many peer entities. Here, we had only 3 weeks but many comparable region-carrier-technology records, so peer comparison was more reliable than long-window temporal modeling. The final run kept 343 cleaned rows, 330 peer-comparison rows, and 39/39 region-week rows. With 6--12+ months of weekly data, the next step would be adding rolling baselines, CUSUM/EWMA, or temporal anomaly models and validating them against known network incidents.
